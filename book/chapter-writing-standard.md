# 章节写作标准

本文件定义 `book/chapters/` 中每一章必须满足的最低教材标准。目标读者是 Rust 零基础、Python/数据科学背景的学习者；章节不能假设读者已经有扎实软件工程基础。

## 每章必须包含的内容

每章都应该尽量包含下面这些部分。标题可以根据章节语义调整，但内容不能缺失。

1. **本章解决什么问题**

   用真实场景说明为什么要学这个概念。不要只说“本章学习 trait”，而要说“为什么因子引擎需要 trait 来表达可替换计算组件”。

2. **学习前提**

   告诉学生需要回忆前面哪些知识。如果没有前提，要明确说“本章可以从零开始”。

3. **Python 对照**

   学生有 Python 经验。必须说明 Python 中相似写法是什么，Rust 为什么更严格，严格性换来了什么。

4. **核心概念逐层解释**

   不能只给定义。要按“直觉 -> 形式 -> 代码 -> 限制 -> 专业判断”的顺序讲。

5. **示例代码走读**

   每章必须指向自己的 `book/examples/chXX-*/`，解释示例中关键函数、类型、测试在证明什么。

6. **动手操作步骤**

   给出可以复制的命令和修改建议。学生应该知道先跑什么、改哪里、错了看什么。

7. **常见错误**

   列出初学者最可能犯的错误，包括编译错误、API 设计错误、性能误判和数值错误。

8. **量化/HPC 连接**

   说明这个概念如何服务高性能量化计算。和主线无关的知识只做必要背景，不展开。

9. **练习**

   练习必须明确修改哪个文件，验证命令是什么，预期学到什么。

10. **本章验收**

    使用“能解释、能实现、能测试、能判断”的形式，而不是只列术语。

## 章节深度要求

每章正文原则上不应少于这些内容量：

- 入门章节：至少 150 行左右，必须有 Python 对照和操作步骤。
- Rust 核心章节：至少 180 行左右，必须有 API 设计判断和示例走读。
- HPC/系统章节：至少 180 行左右，必须有性能或系统设计判断。

行数不是目标本身，但过短通常意味着没有充分解释。

## 代码讲解要求

讲解代码时必须回答：

- 输入是谁拥有的？
- 输出是否分配？
- 错误如何表达？
- 测试覆盖了什么？
- 如果数据规模扩大，哪里可能变慢？
- 如果放进量化系统，边界在哪里？

## 禁止写法

- 只列术语，不解释直觉。
- 只给命令，不解释输出含义。
- 只说“这个更快”，没有 baseline 或测量方法。
- 只说“unsafe 很危险”，不讲安全不变量。
- 大量无关 Web、GUI、业务后端内容。
- 让学生在多个平行目录里找同一章的材料。
