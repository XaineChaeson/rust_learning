# 第 25 章：回测引擎和状态机

因子计算只是研究的一部分。策略研究需要把因子转成信号、持仓、订单、成交、费用和组合权益。回测引擎的核心不是公式，而是状态更新顺序。

## 状态先于指标

一个最小回测系统至少有：

- cash
- position
- market price
- target weight
- order
- trade
- fee
- equity

如果状态更新顺序不清楚，收益、换手、费用、回撤都会错。

## 向量化回测和事件回测

向量化回测快，适合粗略研究。但它容易隐藏成交、费用、延迟、调仓顺序。

事件驱动回测更接近真实系统：

```text
market event -> signal -> order -> fill -> portfolio update
```

学习阶段可以先写单资产 target weight 回测，但必须知道它只是简化模型。

## 手算小样例

专业回测必须有手算测试。比如：

- 初始现金 1000。
- 价格从 100 到 110。
- 全仓持有。
- 最终权益应该是 1100。

这种测试比大数据 smoke 更能发现状态顺序错误。

## 教材化补充：回测错误通常来自状态顺序

很多回测 bug 不是公式错，而是顺序错：

- 先用今天收盘价生成信号，再用同一个价格成交，是否未来函数？
- 手续费是在成交时扣，还是日末扣？
- 调仓前还是调仓后计算权益？
- 现金不足时订单如何处理？

这些问题必须通过状态机和手算测试明确。

## Python 对照

pandas 向量化回测很快，但它容易让时间顺序变模糊。Rust 写事件驱动回测时，顺序必须写成代码：

```text
price -> signal -> target -> trade -> fee -> portfolio -> equity
```

这更啰嗦，但更接近真实执行系统。

## 手算测试的重要性

大规模历史数据无法告诉你某一步状态是否正确。一个两行数据的手算测试更有价值：

```text
cash = 1000
price: 100 -> 110
target = 100%
equity: 1000 -> 1100
```

如果这个都不对，跑十年历史数据也没有意义。

## 常见错误

错误 1：把信号和成交放在同一时间点，造成未来函数。

必须明确数据时间戳和执行时间。

错误 2：忽略费用和滑点。

策略在无成本世界里赚钱，不代表真实可交易。

错误 3：只输出最终收益。

需要 equity curve，才能算回撤、波动、风险和调试状态。

## 代码走读与手算路径

看输入：

```rust
pub struct BacktestRow {
    pub price: f64,
    pub target_weight: f64,
}
```

每一行表达某个时间点的市场价格和目标权重。

看状态：

```rust
let mut cash = initial_cash;
let mut quantity = 0.0;
```

回测不是单纯 map 输入到输出，而是状态逐步更新。

核心步骤：

1. 用当前价格计算当前 equity。
2. 根据 target weight 计算目标持仓价值。
3. 更新 quantity。
4. 更新 cash。
5. 记录新的 equity。

操作清单：

1. 用纸手算两行数据。
2. 给测试增加 target weight 从 1.0 到 0.0。
3. 增加手续费参数。
4. 观察手续费如何改变 cash 和 final equity。

专业判断：回测代码必须先保证小样例正确，再谈大规模性能。

## 自测与复盘问题

1. 回测中的状态至少包括哪些？
2. 为什么手算小样例比大规模 smoke test 更能发现状态 bug？
3. 向量化回测容易隐藏哪些执行细节？
4. 费用和滑点应该在哪个状态更新阶段处理？
5. 为什么只输出最终收益不够？

如果这些问题回答不出来，先画出一条订单从信号到成交再到权益更新的路径。
