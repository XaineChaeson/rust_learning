# 第 22 章：并行计算、chunking 和归约

量化计算天然有很多并行维度：资产、日期分区、参数组合、Monte Carlo path、任务 DAG。并行设计的难点不是启动线程，而是切分任务、合并结果、避免共享热状态，并证明结果确定。

## 可并行的任务形状

适合并行：

- 每个资产独立计算 rolling 因子。
- 每组参数独立跑回测。
- 每条 Monte Carlo path 独立模拟。
- 每个文件分区独立解析。

不适合直接并行：

- 强依赖前一步状态的递推。
- 每个元素都要更新同一个全局 map。
- 输入太小，线程调度成本超过收益。

## chunking

chunk 太小，调度开销大；chunk 太大，负载不均衡。专业系统会根据数据规模和每个任务成本选择 chunk size。

在 rolling 计算中，按时间切分还要处理窗口边界 overlap。按资产切分通常更简单，因为资产之间独立。

## partial result + merge

高性能并行的常见模式：

1. 每个 worker 在本地计算 partial result。
2. 不在热循环里抢锁。
3. 最后单线程或树形归约 merge。

这比 `Arc<Mutex<Vec<_>>>` 更可控。

## 确定性

并行会打乱执行顺序。你必须决定输出是否需要保持顺序。

- 如果需要顺序，结果要携带 index。
- 如果不需要顺序，测试不能依赖 Vec 顺序。
- 浮点归约顺序不同，结果可能有微小差异。

## 教材化补充：并行的单位必须选对

并行前先问：什么东西彼此独立？

量化中常见独立单位：

- 不同 asset。
- 不同 parameter set。
- 不同 simulation path。
- 不同 date partition。

如果你按错误单位切分，会遇到边界依赖。例如 rolling 按时间切分时，每个分区开头需要前一个分区的窗口历史；按资产切分通常没有这个问题。

## Python 对照

Python 里你可能用 multiprocessing 或 joblib 并行参数搜索。问题是进程间传输和序列化成本很高。Rust 多线程能共享只读内存，但你仍然要避免锁竞争。

## chunk size 的直觉

chunk 太小：

- 调度成本高。
- 任务管理开销大。

chunk 太大：

- 某些线程很忙，其他线程空闲。
- 负载不均衡。

好的 chunk size 取决于每个任务的计算量和数据大小，不能拍脑袋。

## 常见错误

错误 1：共享一个全局 Vec，让所有线程 push。

这需要锁，会降低并行收益。

错误 2：输出顺序不稳定却用 `assert_eq`。

如果顺序重要，结果必须携带 index；如果不重要，测试要排序或比较集合。

错误 3：小任务过度并行。

并行有启动、调度、同步成本。小数据常常单线程更快。

## 代码走读与实验设计

看单线程 baseline：

```rust
pub fn sum_assets_single_thread(assets: &[Vec<f64>]) -> Vec<f64>
```

它借用 assets，不取得所有权。

看并行版本：

```rust
pub fn sum_assets_parallel(assets: Vec<Vec<f64>>) -> Vec<f64>
```

它取得 ownership，因为每个 `Vec<f64>` 会被 move 到线程中。这个签名就是并发设计的一部分。

线程里：

```rust
thread::spawn(move || values.iter().sum::<f64>())
```

`move` 表示闭包拿走 `values` 的所有权。这样线程不需要借用主线程栈上的数据。

实验设计：

1. 生成 10 个资产，每个 100 个价格。
2. 生成 10 个资产，每个 1_000_000 个价格。
3. 比较单线程和多线程。
4. 观察小数据并行是否可能更慢。

专业判断：并行适合“每个任务足够重”的场景。不要为了使用线程而使用线程。

## 专业判断清单

并行前先判断任务形状：

1. 单个任务是否足够重，能覆盖线程调度成本？
2. 数据切分是否会引入 overlap 或边界修正？
3. 每个分片能否独立计算 partial result？
4. 合并阶段是否确定、可复现、可测试？
5. 并行后是否增加了内存峰值或缓存 miss？

高性能开发者不会默认相信“多线程一定更快”。你必须保留单线程 baseline，用 benchmark 证明并行版本在目标数据规模上真的更好。
