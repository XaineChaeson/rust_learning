# 第 0 章：如何使用这本教材

你学习 Rust 的目标不是“能看懂语法”，而是能开发高性能量化计算框架。这个目标决定了学习方式：你不能只读，也不能只刷题。你需要在每一章里完成一个闭环：

1. 理解概念。
2. 运行示例。
3. 修改代码。
4. 写测试。
5. 解释错误。
6. 复盘设计。

Rust 对新手不友好的地方，不是语法复杂，而是它会强迫你提前说清楚很多 Python 默认帮你隐藏的东西：数据归谁所有、函数会不会修改输入、失败如何表达、内存是否会被拷贝、并发时谁能访问共享状态。

这正是高性能开发需要的能力。

## 你不是在学一门“更严格的 Python”

Python 的优势是表达快，生态强，适合探索。你可以在 notebook 里快速写出：

```python
import numpy as np

x = np.array([1.0, 2.0, 3.0])
print(x.mean())
```

这个写法隐藏了很多细节：

- `x` 的内存在哪里？
- `mean` 是否分配了新内存？
- `x` 是否连续？
- dtype 是什么？
- 空数组怎么处理？
- NaN 怎么传播？
- 这个计算是否释放 GIL？
- 是否用了 SIMD？

Rust 会逼你面对这些问题。刚开始这会让你觉得慢，但长期看，这是你成为高性能开发者的训练。

## 学习材料的三层结构

这个仓库有三层。

第一层是 `book/`。它像教科书，负责讲知识。你每天主要读这里。

第二层是 `book/examples/`。它放章节配套代码。示例不是项目，是为了让你验证概念。

第三层是 `projects/`。它放阶段项目。项目才是你最终能力的证明。

不要混用这三层：

- 读不懂概念，看 `book/`。
- 想运行最小代码，看 `book/examples/`。
- 想做阶段成果，看 `projects/`。

## 每章怎么学

每章按下面顺序使用。

1. 先读“本章解决什么问题”。
2. 再读 Python 对照，建立迁移关系。
3. 运行章节示例。
4. 修改示例，让它故意失败。
5. 阅读编译器错误。
6. 修复示例。
7. 完成本章练习。
8. 写测试。
9. 写复盘。

复盘可以很短，但必须写。格式：

```text
本章我学到：
Python 中对应习惯：
Rust 中不同点：
我遇到的编译器错误：
我现在还不稳定的概念：
下一步：
```

## 为什么要重视编译器错误

Rust 编译器不是敌人。它像一个严格的代码评审者，在你运行程序之前指出潜在问题。

新手最常见的错误是看见一屏报错就慌。正确方法是：

1. 只看第一条错误。
2. 找到文件和行号。
3. 看 `help` 或 `note`。
4. 问自己：编译器在保护什么？

比如你把一个 `String` 移动后再使用，编译器报错，不是因为它挑剔，而是因为它阻止你使用一个已经不再归你所有的值。这和高性能计算有关：Rust 希望不用垃圾回收也能安全释放内存。

## 高性能路线的学习顺序

你会先学看起来“不高性能”的东西：

- Cargo
- 函数
- `Vec`
- slice
- 所有权
- `Option`
- `Result`
- 测试

这些不是低级内容。它们是后面所有高性能代码的基础。一个 rolling 因子引擎的 API 是否高效，往往取决于你是否理解 `Vec<f64>` 和 `&[f64]` 的区别。

## 本书的第一目标

第一阶段不是写出最快 rolling mean，而是写出这样一个函数，并能解释每个设计：

```rust
pub fn rolling_mean(values: &[f64], window: usize) -> Result<Vec<f64>, NumericError>
```

这里已经包含很多 Rust 核心思想：

- `values: &[f64]` 表示只读借用，不拷贝输入。
- `window: usize` 表示索引和长度类型。
- `Result` 表示可能失败。
- `Vec<f64>` 表示输出由函数新创建并返回。
- `NumericError` 表示错误是类型，不是随便一段字符串。

当你能自然写出并解释这个函数，才真正进入了 Rust 量化计算的门。

## 教材化补充：你应该怎样使用一章

如果你以前主要写 Python 数据分析代码，很容易把学习过程理解成“读懂一段代码，然后在 notebook 里改几行”。Rust 不是这样学的。Rust 的反馈回路更像工程训练：你写出一个明确的接口，编译器检查所有权、类型和生命周期，测试检查行为，benchmark 检查性能。

每一章都应该按这个顺序走：

1. **先读概念**：不要急着复制代码。先知道这个概念解决什么问题。
2. **跑原始示例**：确认你当前环境和课程代码是正常的。
3. **做小改动**：只改当前章节的 `book/examples/chXX-*/`。
4. **读错误信息**：编译器错误不是失败，而是 Rust 训练你理解边界的方式。
5. **写测试或改测试**：没有测试，就不知道自己是否真的理解。
6. **写复盘**：用自己的话解释 Python 写法和 Rust 写法有什么差异。

一个最小学习循环如下：

```bash
cargo test -p ch02-values-types-functions
```

如果测试通过，改一行代码，让测试失败；再修复它。这个过程比单纯阅读更重要。

## 零基础学生最容易误解的地方

第一个误解：把编译错误当成阻碍。Rust 编译器确实严格，但它指出的问题往往是 Python 会在运行很久之后才暴露的问题，例如空输入、类型不清、数据被意外修改、并发共享不安全。

第二个误解：把所有权理解成“语法规则”。所有权不是语法游戏，它是 Rust 不需要 GC 也能安全释放资源的基础。你后面写高性能计算时，能否避免无意义复制，很大程度上取决于是否理解所有权。

第三个误解：以为性能优化从 unsafe 或 SIMD 开始。专业优化从 baseline、测试、benchmark 和数据布局开始。unsafe 是最后的边界工具，不是第一反应。

## 学完一章的最低标准

你不需要一章内记住所有细节，但必须能做到：

- 能复述本章解决的问题。
- 能运行本章示例命令。
- 能指出示例里最关键的函数或类型。
- 能解释输入是否拥有数据，输出是否分配。
- 能写一个小测试证明自己改动是对的。

只有做到这些，才算真正完成一章。
